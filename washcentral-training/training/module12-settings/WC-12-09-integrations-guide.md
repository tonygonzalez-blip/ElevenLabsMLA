# WC-12-09 · Integrations — SSO, Accounting, and Help Desk (Settings)

v1.0 · July 9, 2026 · Author: Claude — Micrologic training project
**Verified against demo.washcentral.com on July 9, 2026.**

## Purpose

This lesson teaches you to read a WashCentral integration page without touching it. Every integration in Settings follows the same layout, so once you can read one you can read all of them. We go deep on **Zoho Desk**, the help-desk integration, then walk the single sign-on providers (**Microsoft Entra ID**, **Google Workspace SSO**, **Okta**, **Amazon Cognito**), user provisioning (**SCIM Provisioning**), and the two accounting connectors (**Sage Accounting**, **QuickBooks Online**) at the same anatomy level. The whole lesson is view only. You never enter credentials, and you never click **Test Connection**.

## Audience and prerequisites

Internal Micrologic technicians who already know how to open Settings and read a settings page (WC-12-01). You should be signed in. No integration experience is assumed. This is a support-reading skill: you learn what each page shows so you can tell a client whether an integration is connected, not how to configure one. Configuring, testing, and editing credentials are administrator jobs and are out of scope here.

## Navigation path

Each integration has its own page under Settings. The breadcrumb reads `Settings › [integration name]`, and the Settings left rail carries an **Integrations** entry at the bottom that these pages sit under. So the path is `Settings → Integrations → Zoho Desk` (or any other integration). The landing page that lists all eight integrations as a set was not reachable in this session's corpus, so the exact hub layout is TBD — parked (see Notes). You reach each integration page directly, and the breadcrumb confirms where you are.

## Steps

One action per step. Hold each new page for two to three seconds before moving on. The whole walk is read-only.

1. Open the **Zoho Desk** integration page under `Settings → Integrations`. **Expected:** the breadcrumb reads `Settings › Zoho Desk` and the Settings left rail shows **Integrations** at the bottom. This is the page every other integration is a variation of, so read it slowly.
2. Read the header identity block. **Expected:** the title **Zoho Desk** sits beside a **Help Desk Integration** badge and a **⚠ Not Configured** status pill, with the subtitle **Support Tickets · Settings** underneath. The badge tells you what the integration is for; the pill tells you it is not connected yet in this tenant.
3. Read the three method chips under the title. **Expected:** **OAuth 2.0**, **Live View**, and **Server-side Token**. These name how WashCentral would talk to Zoho once connected. On every integration page the chips change to match that vendor, and they are the fastest way to see the connection method at a glance.
4. Find the two buttons at the top right: **Test Connection** and **Edit Configuration**. **Expected:** both are visible and both are administrator actions. Do not click either. **Test Connection** would fire a live call out to the vendor, and **Edit Configuration** opens the credential editor. As a technician you read this page; running or editing it is not your job and is not part of this lesson. **[READ-ONLY]** — locate both buttons, click neither.
5. Read the four status tiles across the top of the page. **Expected:** **Connection** (Not set), **Data Center** (US), **Org ID**, and **Sync Direction** (both). This strip is the health summary. **Connection** reading Not set lines up with the **⚠ Not Configured** pill above, so the two agree: nothing is wired up yet.
6. Note the two tabs below the tiles. **Expected:** **Settings** (open) and **Instructions**. **Settings** shows the live field values you are reading now. **Instructions** holds the vendor setup steps; its contents were not captured in this session's corpus, so what the Instructions tab shows is TBD — parked (see Notes).
7. On the **Settings** tab, read the **CREDENTIALS** field group by its labels only. **Expected:** **CLIENT ID**, **CLIENT SECRET**, **ORG ID**, **DATA CENTER**, **ACCESS TOKEN**, **REFRESH TOKEN**, **TOKEN EXPIRY**, and **ACCOUNTS SERVER**. These hold the secrets that authorize the connection. You never read, speak, or capture credential values; in this demo tenant the secret fields sit empty (Not set / —), and the values themselves are out of scope. **[READ-ONLY]**
8. Read the operational field groups below **CREDENTIALS**. **Expected:** **GENERAL** (**ENABLED**, **ADMIN NOTES**), **ROUTING** (**DEFAULT DEPARTMENT**, **DEFAULT ASSIGNEE**), **SYNC** (**SYNC DIRECTION**, set to both), **BEHAVIOR** (**AUTO-CREATE CONTACTS**), and **MAPPING** (**STATUS MAPPING**, **SLA MAPPING**). These decide how support tickets route and sync once the integration is live. That is the full Zoho anatomy; the rest of the lesson maps it onto the other integrations.
9. Open the **Microsoft Entra ID** integration to see the single sign-on version of the same page. **Expected:** the same skeleton with a **SSO Integration** badge, the **⚠ Not Configured** pill, the subtitle **Single Sign-On · Settings**, and method chips **Authentication**, **Microsoft 365**, **MSAL.js**. The four tiles change to **Connection**, **Issuer**, **Client ID**, and **Scopes**. The field groups cover **CONNECTION**, **PROVISIONING**, **MAPPING**, and **AUTH**, and there is a **REDIRECT URI** field with a **Copy** button to paste into the identity provider.
10. Glance across the other three SSO providers: **Google Workspace SSO**, **Okta**, and **Amazon Cognito**. **Expected:** the layout is identical to **Microsoft Entra ID**. Only the provider name and the method chips differ: **Google Workspace SSO** shows **Authentication / Google Workspace / OAuth 2.0**, **Okta** shows **Authentication / OIDC / PKCE**, and **Amazon Cognito** shows **Authentication / AWS / PKCE / OAuth 2.0**. Same four tiles, same **⚠ Not Configured** state, same **Copy**-able redirect URI. Learn one SSO page and you have learned all four.
11. Open **SCIM Provisioning**. **Expected:** a **User Provisioning** badge, the subtitle **SCIM 2.0 · Settings**, chips **Auto-Provisioning / Microsoft Entra ID / REST API**, and a **← SSO Settings** back link. The tiles read **Status** (Not Set Up), **SCIM URL**, **Bearer Token** (Not Set), and **Last Event**. The field groups are **ENDPOINT CONFIGURATION** (a **FULL TENANT URL** with a **Copy** button, and a **BEARER TOKEN**) and a **SYNC LOG** that is empty until provisioning events arrive. This is the page that automates creating and deactivating users from your identity provider.
12. Open **Sage Accounting** to see the accounting shape. **Expected:** an **Accounting Integration** badge, the subtitle **General Ledger · Settings**, chips **Accounting / ERP / XML API**, and tiles **Connection**, **Environment** (Production), **Company ID**, and **Sync Schedule** (0 2 * * *). The groups add accounting-specific fields: **SYNC** carries **SYNC SCHEDULE (CRON)**, **BATCH SIZE**, and **MAX RETRIES**, and a **SAFETY** group carries **DRY RUN**. The **Sync Schedule** tile shows a cron value because these connectors post on a timer, not on every sale.
13. Open **QuickBooks Online**, the other accounting connector. **Expected:** the same **Accounting Integration** badge and **General Ledger · Settings** subtitle, chips **Accounting / OAuth 2.0 / REST API**, and tiles **Connection**, **Environment**, **Realm**, and **Token**. Its **CREDENTIALS** group names a **REALM / COMPANY ID** alongside the OAuth tokens. Sage and QuickBooks do the same job through different vendors, which is exactly the pattern you saw across the four SSO providers.
14. Recap the read and run the three-step check. **Expected:** every integration page is the same shape — an identity block with a badge and a **⚠ Not Configured** or connected status, method chips, **Test Connection** and **Edit Configuration**, four status tiles, **Settings** and **Instructions** tabs, and grouped fields. When a client says an integration is not working, you read three things in order: the status pill, then where **Test Connection** sits (an administrator runs it, not you), then **TOKEN EXPIRY** and the credential state. You can answer "is it connected?" from this page without filing a bug.

**Exercise 12.9a [READ-ONLY]:** a client reports the Help Desk tab shows no tickets. On the **Zoho Desk** page, name the three things you check, in order, before escalating: (1) the **⚠ Not Configured** status pill, (2) the **Test Connection** button (note who runs it), and (3) **TOKEN EXPIRY** in the **CREDENTIALS** group. Do not click **Test Connection** and do not open **Edit Configuration**. The connected ticket view itself lives in the Help Desk lesson (WC-10-03), and the per-user SSO status lives in Users & Security (WC-12-06).

## Expected results

You can open any integration page in Settings and read it cold: what the integration is for (the badge), whether it is connected (the status pill and the **Connection** tile), how it authenticates (the method chips), and how it behaves once live (the field groups). You can tell the four SSO providers apart by their chips while knowing the page is otherwise the same, and you can do the same for the two accounting connectors. You never enter a credential and never click **Test Connection**, and you can still answer a client's "is this integration working?" from what the page shows.

## Notes and troubleshooting (observed behavior only)

- Every integration page in this lesson shows the **⚠ Not Configured** status in the demo tenant. That is the not-connected baseline. A connected page would show a different status and a live **Connection** tile.
- The **CREDENTIALS** group holds secrets. You read the field labels to understand the page, never the values. In this demo tenant the secret fields are empty (Not set / —), so there is nothing to mask here, but the rule holds everywhere: credential values never go into a note, a screenshot, or a ticket.
- **Test Connection** and **Edit Configuration** are administrator actions. This lesson locates both and clicks neither. **Test Connection** would reach out to the live vendor, and **Edit Configuration** opens the credential editor; its contents are TBD — parked (not opened in this session).
- The **Instructions** tab on each page holds vendor setup steps. Its content was not captured in this session's corpus, so it reads TBD — parked.
- The landing page that lists all eight integrations as a set returned a 404 in this session's corpus, so the hub layout is TBD — parked. You reach each integration page directly, and the breadcrumb (`Settings › [name]`) confirms the location.
- Once an integration is connected, its effect shows up on other pages, not here: help-desk tickets appear in the Help Desk area (lesson WC-10-03), and SSO turns on per user in Users & Security (lesson WC-12-06). Those connected states were not observable from the integration pages in this corpus.
- The SSO pages expose a redirect/callback URL with a **Copy** button, and **SCIM Provisioning** exposes a **FULL TENANT URL** the same way. The exact URL strings are environment-specific; copy them from the live page rather than from any document.
- A "Still there?" session-timeout dialog can appear on these pages after idle time. It is not part of this lesson and is covered in WC-01-03; the timeout specifics are parked (PARKED-ITEMS.md #9).
