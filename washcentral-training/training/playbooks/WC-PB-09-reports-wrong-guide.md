# WC-PB-09 · Reports wrong or missing (Troubleshooting Playbook)

v1.0 · July 9, 2026 · Author: Claude — Micrologic training project
**Verified against demo.washcentral.com on July 9, 2026.**

## Purpose

A site manager says a report shows the wrong numbers. Or the report comes back empty. Or the daily report that used to land in someone's inbox stopped arriving. This playbook is the order you work those tickets. You name the symptom first, then follow one of two trails: a wrong-or-empty report almost always comes down to the filter bar, and a report that never arrived is a scheduling question. You read at every stop and change nothing. The fixes (adjusting a schedule, correcting a task) happen in production with the right role, never from the demo tenant.

Two areas carry the whole playbook, both from the left navigation rail: **Reports**, which holds the reports hub and the **Report Viewer**, and **Settings → Automation → Task Scheduler**, which holds the job that sends a scheduled report. Every screen here is read-only.

## Audience and prerequisites

Internal Micrologic technicians on Tech Support and Onboarding who take reporting tickets. You should be signed in and comfortable with the global header and the left navigation rail (WC-01-01 and WC-01-02). No reporting background is assumed. This is a diagnosis-and-escalation playbook, so you read filters, views, and run times, then route. You do not schedule, export, or edit a report or a task here.

## Navigation path

The **Reports** module opens from the left navigation rail. Inside it, a **VIEWS** panel and a **CATEGORIES** panel run down the left, and opening any report loads the **Report Viewer**. The scheduling side lives under `Settings → Automation → Task Scheduler`, reached from the Settings gear at the bottom of the rail.

## Steps

One action per step. Hold each new screen two to three seconds before moving on. Nothing in this playbook is scheduled, exported, created, or edited.

1. Open **Reports** from the left navigation rail and read the **VIEWS** panel and the **Report List**. **Expected:** the page title reads **Reports**. Down the left, **VIEWS** lists **All Reports** (173), **Favorites** (5), **Scheduled Reports** (3), **Advanced Analytics** (23), and **Custom Builder**, with a **CATEGORIES** panel below it (Financial 8, Traffic & Sales 4, Revenue 6, and so on). The main panel opens on **All Reports**, subtitle "Browse and run all standard and custom reports", above the **Report List** with columns **★ · Report Name · Category · Type · Last Run**. The **Type** column tags each report **Standard**, **Scheduled**, or **Analytics**. This one hub is where every standard and custom report a site runs is listed.
2. Name the symptom before you touch a filter. **Expected:** one of three. Wrong figures on a report the manager just ran, or an empty report, sends you to the **Report Viewer** and its filter bar (steps 3 to 5). A scheduled report that never arrived sends you to the **Scheduled Reports** view and then the scheduler (steps 6 to 8). A "why don't these two numbers match" question about a demo tile is usually not a bug at all (step 5). Deciding this first keeps you from re-running the wrong report ten times.
3. For the wrong-or-empty trail, open the report from the **Report List** to load the **Report Viewer**. **Expected:** the report opens in the Report Viewer with a filter bar across the top. *(Source: DISCREPANCIES.md #24, July 8 verified walk.)* The bar carries **More Filters ▾**, a location selector reading **All Locations**, a **Date Range** with **From** and **To**, a **✕ Close Report** control, and a **▶ Run Report** control. The **Schedule Report** button is on the reports hub, not in the viewer. The Report Viewer itself needs a report-open parameter and was not captured in the July 9 corpus, so its rendered results are TBD — parked (see Notes); the filter-bar labels are grounded in DISCREPANCIES.md #24.
4. Read the filter bar, correct the scope, then re-pull with **▶ Run Report**. Do not click **Schedule Report** or any export control. **Expected:** the report re-runs read-only against the scope you set. Most "wrong numbers" tickets are a scope mismatch: the location selector was left on one site when the manager meant the whole network, or the **Date Range** **From**/**To** missed the days in question. Set the location and the dates the ticket actually asks about, run it again, and compare. A report is only wrong once its filters are right. **[READ-ONLY]**
5. Rule out a demo-tenant number mismatch before you file anything. **Expected:** in this demo tenant, a KPI tile and the table beneath it frequently disagree (per DISCREPANCIES.md, demo-data observations: Customers KPI 2,418 vs table 1,643,807, and similar). That is a known demo artifact, not a platform defect. Do not open a numbers bug from the demo tenant; reproduce it in production first. This step is the difference between a real report ticket and noise.
6. For the report-never-arrived trail, open the **Scheduled Reports** view in the **VIEWS** panel. **Expected:** the view is labeled **Scheduled Reports** with a count of **3**. Confirm the report the manager is missing is actually one of the scheduled reports and read its **Last Run**. In the **All Reports** list you can already see scheduled reports carry the **Type** chip **Scheduled** with a real **Last Run** timestamp (for example, **Daily, Period, and Shift Financial 2.0** last ran **03-25-2026 9:14 AM**). The filtered contents of the **Scheduled Reports** view were not separately captured in the July 9 corpus, so the exact three rows and any per-report download control are TBD — parked (see Notes). If the report is not scheduled at all, that is your answer.
7. Cross-check the job that sends it: go to **Settings → Automation → Task Scheduler** and read the summary strip. **Expected:** the page title reads **Task Scheduler** under **Settings › Task Scheduler**, with the **Automation** group open in the settings nav (**Task Scheduler**, **Task Parameters**, **Mass SMS Log**, **User Levels**, **Integrations**). The KPI strip reads **18 Total Tasks · 16 Active · 2 Running Now · 0 Failed**, and three health panels (**Task Scheduler Trend**, **Task Distribution**, **Task Status**) sit above the list. **Failed** at **0** tells you no scheduled job errored out; a non-zero **Failed** count is your first lead when a scheduled report goes missing.
8. Read the report job's row in the **Task Scheduler List**. Change nothing. **Expected:** the list header reads **Task Scheduler List**, and its columns are **Task Name · Schedule · Last Run · Next Run · Status**. The report job is **Daily Report Email**, showing **Daily 6:00 AM · Today 6:00 AM · Tomorrow 6:00 AM · Active**. Read across the row: the **Schedule** is when it should send, **Last Run** whether it ran, **Next Run** when it goes again, and **Status** whether it is **Active**. A report that never arrived usually shows here as a stale **Last Run** or a status that is not **Active**. You read this row and route it; you never click **+ New Record** and you never edit a task. **[READ-ONLY]**
9. Package the escalation from what you read, and stop. **Expected:** a ticket that names the report, the location scope and the **Date Range** you ran it with, and, for a scheduled report, the job's **Last Run**, **Next Run**, and **Status** from **Task Scheduler**. Attach a screenshot. Leave demo-tenant KPI-vs-table complaints out of it. Diagnosis and routing are your job on this playbook; changing a schedule or re-sending a report happens in production with the right role. **[READ-ONLY]**
10. Recap the path. **Expected:** you named the symptom, checked the **Report Viewer** filter bar for a wrong or empty report, checked the **Scheduled Reports** view and the **Daily Report Email** job in **Task Scheduler** for a missing one, ruled out a demo-data mismatch, and escalated with the report, its filters, and its run times. Every screen was read-only.

## Expected results

You can tell a wrong-numbers ticket from a missing-report ticket before you navigate. You know the wrong-or-empty trail runs through the **Report Viewer** filter bar (**All Locations** scope and **Date Range** **From**/**To**), and the missing-report trail runs through the **Scheduled Reports** view and the **Daily Report Email** job in **Settings → Automation → Task Scheduler** (its **Last Run**, **Next Run**, **Status**, and the **Failed** KPI). You rule out demo-tenant KPI-vs-table mismatches instead of filing them, and you escalate with the report name, the filters, and the run times without changing a thing.

**Knowledge check:** (1) A manager says a report's totals are too low. What is the first thing you read before disputing the numbers? (2) Where do you confirm a report is actually scheduled, and where do you confirm the job that sends it ran? (3) Which single KPI on the Task Scheduler tells you a scheduled job errored out? (4) Two numbers on a demo dashboard disagree. Do you file a bug? Why or why not?

## Notes and troubleshooting (observed behavior only)

- The **Report Viewer** filter bar controls (**More Filters ▾**, **All Locations**, **Date Range** **From**/**To**, **✕ Close Report**, **▶ Run Report**) are grounded in DISCREPANCIES.md #24 from the July 8 verified walk. The viewer needs a report-open parameter and was not captured in the July 9 corpus, so the rendered report body is TBD — parked. Confirm the filter bar and re-run against the live viewer at record time.
- The **Scheduled Reports** view shows a count of **3** in the July 9 corpus, but the digest captured the default **All Reports** view, not the filtered view. The exact three scheduled rows, and the legacy "download from Latest Result" control the plan references (legacy B.29), are TBD — parked. What is grounded: scheduled reports carry the **Type** chip **Scheduled** and a real **Last Run** in the **All Reports** list.
- On **Task Scheduler** the summary reads **18 Total Tasks** and the list card reads **18 records**, but the table shows only **Showing 1–3 of 3 records** (**Daily Report Email**, **SMS Campaigns**, **Database Backup**). The report job **Daily Report Email** is one of the three that loaded. Treat the count mismatch as a demo-tenant data artifact, the same class as the KPI-vs-table gaps in DISCREPANCIES.md; confirm the full task list on the live tenant before telling anyone a job is missing.
- The **Failed** KPI reads **0** in the July 9 capture, so there is no failed report job to inspect here. What a failed row looks like (its status text and any error detail) is TBD — parked until a job is observed in a failed state.
- KPI-vs-table mismatches inside the demo tenant are a known artifact (DISCREPANCIES.md, demo-data observations). Do not file demo-data number bugs from the demo tenant. Reproduce in production first.
- **Task Parameters** (the sibling page under **Automation**) returned "Couldn't load task parameters (Failed to fetch)" in the July 9 capture. It is out of scope for this playbook and noted only so an empty page there is not mistaken for a reporting fault.
- Both pages also show the session idle prompt ("Still there?" with a 2:00 countdown) and the **Ask Shyne** header button. Both are out of scope here; session behavior is parked (PARKED-ITEMS.md #9).
